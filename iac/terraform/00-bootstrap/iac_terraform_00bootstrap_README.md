# 00-bootstrap (Terraform)

## Why this directory has no Terraform code

The Terraform state backend (S3 bucket + DynamoDB lock table + KMS key) and the
GitHub Actions OIDC trust must exist *before* any Terraform layer can run.
Bootstrapping these resources via Terraform creates a chicken-and-egg problem:
the state for the backend would need to be stored somewhere, and that somewhere
is what we're trying to create.

The industry-standard pattern is to provision these resources once, manually,
via a script. Subsequent IaC for every other layer uses the resulting backend.

## How it was bootstrapped

Run from the repository root, with admin credentials in the AWS Management
account:

```powershell
$env:AWS_PROFILE = "bootstrap"
python scripts\bootstrap.py
```

Source: [`scripts/bootstrap.py`](../../../scripts/bootstrap.py)

## Resources created in the Management account

| Resource | Name | Purpose |
|---|---|---|
| S3 bucket | `ai-sec-analyst-tfstate-<account-id>` | Terraform state, versioned + KMS-encrypted, TLS-only |
| DynamoDB table | `ai-sec-analyst-tflocks` | Terraform state locking |
| KMS CMK | `alias/ai-sec-analyst-tfstate` | State encryption, automatic rotation enabled |
| OIDC provider | `token.actions.githubusercontent.com` | GitHub Actions federation |
| IAM role | `gha-bootstrap-role` | Assumed by GitHub Actions for IaC deployments |

## Backend configuration

Generated by the script and committed at `iac/terraform/backend.hcl`. Each
layer's `terraform init` references it:

```bash
terraform init \
  -backend-config=../backend.hcl \
  -backend-config="key=01-foundation/terraform.tfstate"
```

## Recreating from scratch

If the bootstrap resources are ever destroyed:

1. Recreate IAM access keys for `bootstrap-admin` (or use a fresh admin user).
2. Run `python scripts\bootstrap.py` again — the script is idempotent.
3. Delete the access keys.
4. Re-init each Terraform layer (`terraform init -reconfigure`).

State data itself will be lost, so prefer to never destroy `00-bootstrap`.
